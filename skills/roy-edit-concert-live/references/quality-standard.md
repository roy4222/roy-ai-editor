# Concert clip quality standard

Use HACHI Birthday LIVE 2025 `万華鏡` v3 as the first golden reference. The 2026-07-12 prototype is Roy's approximate 80/100 baseline, not the target ceiling.

Use a versioned Concert Subtitle Profile rather than per-project style constants. Record the profile ID/version in the Render Manifest. Profile changes require representative golden-fixture A/B and burned-pixel regression before Roy promotes a new version; they never silently rewrite prior renders. In Source Japanese Subtitle Mode preserve the source Japanese styling and apply only the profile's compatible Traditional Chinese and Safe-Area Recovery rules.

Run the complete Concert Golden Corpus for Production Capability, model, adapter, and profile promotion. Public CI uses only synthetic Test Fixtures; private real clips/annotations stay under the 5 GiB RoyMedia shared Retention Hold and cover source-subtitle modes, multi-singer cases, alignment stressors, theme-aware translation, and ruby ambiguity. A single successful production video cannot waive a corpus regression.

Run every Media Project under one pinned Production Toolchain Release and record its exact tool/model/profile checksums in the Render Manifest. A Maintenance Train candidate must pass CI, the complete corpus, Mac host smoke, restart/resume, and rollback before activation. Only candidates whose normalized text, timing, pixels, media properties, decisions, schemas, and security scope remain equivalent may auto-promote; any governed-output difference requires one A/B Toolchain Promotion Review. Never change the release inside a running job, and retain the last-known-good release for rollback.

Concert V1 release requires all spikes/corpus checks plus three representative real Media Projects and at least 12 Track Jobs: single/no-source-Japanese, multi-singer/complete-source-Japanese including Safe-Area Recovery, and cover/theme/ruby difficulty. Exercise host restart, stage retry, duplicate URL, upload retry, Publish Verification, Retention Plan dry-run, and Recovery Vault restore. Any silent text/ruby/order/boundary/media error, manual JSON/file/shell repair, required editing GUI, or duplicate remote upload fails acceptance.

Exercise the Host Health Supervisor with missing/wrong RoyMedia, low space, killed worker, stale lease, corrupt reproducible cache, toolchain checksum drift, network loss, quota exhaustion, revoked OAuth, channel mismatch, and stale/failed Recovery Vault. Each case must block only affected stages, preserve a secret-free Host Health Snapshot, perform no action outside the Safe Self-Healing Action allowlist, avoid duplicate work/uploads, and create at most one deduplicated shared blocker when intervention is truly required.

## Release gates

- **Rights:** Record Intake Authorization, source/channel policy, lyric and translation provenance, attribution, reuse limits, uncertainty, and the Rights Audit route. Explicit prohibitions or access-control bypasses are FAIL; missing or conflicting evidence is Exception Review rather than an automatic pass.
- **Lyrics approval:** Record the official Japanese source, all Bahamut candidates, selected translator/source, reuse terms, performed repeats, proposed display breaks, Ruby Map, packet version, and Roy's explicit approval. Any later text, line-break, or ruby-map change invalidates only the affected packet.
- **Boundaries:** Keep the first note/lead-in and full vocal/instrumental ending. Exclude unintended next-song or MC leakage.
- **Text:** Match the performed lyrics, including repeats. Build and follow the Song Interpretation Brief so the Traditional Chinese preserves the whole song's narrative, theme, emotional arc, imagery, and cross-line meaning rather than translating isolated words mechanically. Use natural Taiwan Traditional Chinese, do not force rhyme/singability at the expense of meaning, consult relevant Roy Translation Notebook cases with provenance, and follow reuse permission.
- **Line breaks:** Default to one approved source line per subtitle event. Never merge three source lines. Merge two only with Roy's explicit approval. Split overlong lines at a sung breath, punctuation, or grammatical boundary and split the translation at the same semantic boundary. Target at most 80% of frame width and fail any clipping or cramped safe-area result.
- **Source subtitles:** Before selecting layout, distinguish soft subtitle streams, burned-in Japanese lyrics, intermittent stage captions, and no subtitles. Preserve source Japanese only at 100% performed-line and approved text/order/timing coverage. Otherwise persist one track-wide Source Subtitle Fallback Plan: disable incomplete soft subtitles and render the full Normal Bilingual Subtitle Mode, or first pass deterministic whole-track Caption Region Replacement for burned text. Replacement requires zero readable old-text residue, full new Japanese/ruby/Chinese coverage, and no material composition harm. If it cannot pass, route an exact skip/version exception; never accept 99%, patch only missing lines, or mix modes.
- **Karaoke:** Align the approved display lines by sung phoneme/mora, not even character timing. Handle rests, breaths, melisma, and long notes. Let the Automated Quality Verdict choose the highest passing Timing Fidelity Tier without fixed Roy approval: degrade failed fine timing to passing line timing, but fail incorrect line boundaries, missing repeats, rewritten text, or crossed line order. For every furigana span, verify the hiragana reading against the approved Ruby Map and center it above that exact kanji-only base span.
- **Ruby evidence:** Auto-pass an official reading, or agreement between two independent Japanese analyzers when no ambiguity flag or conflict with performed audio exists. Names, jukujikun, ateji, lyric-specific readings, analyzer disagreements, and acoustic conflicts require Exception Review; visual placement cannot prove linguistic correctness.
- **Singers:** Follow the Singer Color Policy: use verified official colors with sufficient contrast, keep colors stable, show duets/overlaps deliberately, and use neutral styling rather than guessing uncertain attribution. Preserve source Japanese colors and keep added Chinese neutral in Source Japanese Subtitle Mode; only semantic or metadata-critical identity uncertainty enters Exception Review.
- **Visual QA:** Inspect every original-resolution, numbered, full-width subtitle-band crop from the source audit and the actually burned output MP4. Compare wording and breaks against the approved line map and applicable Ruby Map; separately check source-Japanese coverage, ruby readings/base spans/centering, Chinese pairing/position, clipping, and repeated/final sections. Use Computer Use for a second read-only visual pass through the crops or video and preserve the per-line verdict; a model report alone cannot pass the gate.
- **Media:** Check audio pops/clipping/mutes, black/frozen frames, encoding, resolution, frame rate, duration, and file integrity.
- **Publish:** Use the selected Channel Publish Profile version to generate complete provisional metadata. Before network I/O, fsync a deterministic Publish Intent; keep resumable session URI secret, persist its reference before bytes, and require Orphan Upload Reconciliation before any replacement insert. Exact one-candidate reconciliation adopts; ambiguity blocks. Require Channel Binding Profile matches, private visibility, `notifySubscribers=false`, and durable returned video ID. Publish Verification confirms API processing/playability plus restrictions through isolated `roy-studio-verify` with default-deny actions: unrestricted is PASS; playable claim-only is WARN; blocked, muted, removed, failed, or unplayable is FAIL. Never let the worker update, delete, browser-upload, dispute, or promote.

- **Preregistered gates:** Every candidate, corpus, maintenance, and acceptance run must pin the Concert V1 Quality Registry version/hash and fixture manifest before execution. Universal zero-tolerance failures, sample counts, timing/ruby/pixel/browser/publish/resource thresholds and normalized-output equivalence come only from that registry. A post-result threshold change requires a new version and full rerun; it cannot relabel the old result.
