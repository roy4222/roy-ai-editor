# Concert workflow

1. **Intake:** From the Codex Control Surface, persist the URL, languages, Track Selection intent, supplied timestamps, preferences, Intake Authorization, request ID, and originating Codex task as an immutable Production Job Request. Create the Media Project and Project Manifest before expensive work; the launchd-managed Production Worker acquires a lease and resumes idempotent stages from checkpoints independently of the Codex session. Queue any additional requests; grant only one project a heavy media slot, then parallelize Track Jobs inside it according to the benchmarked resource profile and release the slot while waiting for a human gate. Persist each review/blocker/result in the Review Outbox so a deduplicating Codex monitor can return it to the originating task even after Codex was closed.
2. **Rights:** Treat Roy's submitted URL as Intake Authorization for local processing and private upload. Check the video description, channel/company derivative-work policy, song restrictions, and translation reuse terms; save sources, dates, attribution, and uncertainty. Continue automatically when evidence permits, stop explicit prohibitions or access-control bypasses, and ask Roy only when evidence is missing or conflicting.
3. **Ingest:** Use the CLI when implemented. Save metadata beside the manifest. Keep media outside Git. Never bypass access controls.
4. **Discover tracks:** Run Timeline Discovery and dynamic lyrics research through `roy-concert-discovery` under its Browser Read-Only Enforcement Profile: enforce domain/path allowlist, content boundaries, output limit, default-deny action policy, and Browser Action Ledger; open/read/snapshot/screenshot/scroll or use only allowlisted expand controls. Reject typing, submission, download, eval, social interaction, mutation, and unexpected navigation. Inspect creator chapters/description and creator, pinned, or otherwise credible timeline comments, and preserve author, permalink, text, capture time, screenshot, and ledger hash. Treat Roy's Provided Setlist Timeline as highest priority when present. Only then use audio/visual/lyrics/ASR inference, and route CAPTCHA, reauthentication, access barriers, policy failures, or material conflicts to Exception Review.
5. **Segment:** Combine timeline, waveform, vocals/music/applause/MC, and visual transitions. Preview uncertain endings. Keep the full instrumental decay; silence removal alone cannot choose the end.
6. **Source subtitle audit:** Follow `source-subtitle-audit.md`. Inspect streams and original-source screenshots for every selected track, classify existing Japanese coverage, and preserve the evidence. Do not choose the render layout from one matching frame.
7. **Lyrics/translation approval:** Follow `lyrics-approval-gate.md`. Use official Japanese lyrics, run the required Bahamut searches, and present candidate URLs, translator, reuse terms, full approved-or-user-provided paired text, repeats, uncertainties, the proposed source-line-to-display-line map, the Ruby Map, and source-subtitle matches. Apply the Ruby Evidence Policy first and summarize only unresolved reading exceptions for Roy while preserving the full versioned map. If the bounded search finds no acceptable reusable Traditional Chinese translation, include a clearly labeled, provenance-recorded independent AI draft. Present all selected songs as one Concert Lyrics Review and stop until Roy explicitly approves the exact packet versions.
8. **Align:** Continue only for approved Lyrics Packets and run each as an independent Track Job. Treat the approved lyrics, line map, and Ruby Map as text truth. Normalize approved repeats, optionally isolate vocals, forced-align, refine by phoneme/mora, and handle rests, breaths, melisma, and long vowels. Let the Automated Quality Verdict choose the highest passing Timing Fidelity Tier without a fixed timing approval; fine timing may degrade automatically to passing line timing, while failed line boundaries or text/order changes enter Exception Review. In Source Japanese Subtitle Mode, align the Traditional Chinese to the verified source-Japanese events without rendering duplicate Japanese/ruby; otherwise put each approved reading above its exact kanji-only base span. Invalidate only the affected packet and stop that Track Job if its text, line map, Ruby Map, or source-subtitle classification must change; do not block unrelated passing jobs.
9. **Multiple singers:** Combine cast, lyric assignment, audio, face/position evidence, and Roy hints. Apply verified official member colors with readability adjustments and use a deliberate group style for duet/chorus lines. When attribution remains uncertain, use a neutral color and continue; enter Exception Review only when identity changes lyric interpretation, credits, or formal metadata. In Source Japanese Subtitle Mode preserve source Japanese colors and keep added Chinese neutral rather than inferring singer identity from source styling.
10. **Render/QA:** Use deterministic CLI execution and the Production Job Request's pinned Concert V1 Quality Registry. Compare every rendered line against the approved line map, applicable Ruby Map, Source Subtitle Audit, and Source Subtitle Fallback Plan. Preserve complete source Japanese only in Source Japanese Subtitle Mode; otherwise disable incomplete soft tracks or require a passing whole-track Caption Region Replacement before Normal Bilingual Subtitle Mode. Render a review MP4, run `scripts/karaoke_visual_qa.py` for every lyric line, and crop the original-resolution full-width subtitle band. Inspect every numbered source/output crop directly, then use Computer Use for a second read-only pass. Preserve per-line verdicts for wording, repeats, base spans/readings, Chinese pairing, old-text residue, safe areas, clipping, alignment and full media decode; no unburned preview, coordinates, scaled overview, or model report can pass the gate.
11. **Publish/retain:** Generate media, subtitles, provenance, credits, metadata, and QA. When every required Automated Quality Verdict is PASS, record a Verified Render and provisional metadata/thumbnail. Before network I/O, fsync a deterministic Publish Intent. Initiate a resumable Publish Attempt, store the session URI only in Keychain/encrypted private state, and persist its opaque reference before media bytes. On any uncertain outcome, query the stored session and run exact-marker own-channel Orphan Upload Reconciliation; adopt one unique match, fail closed on ambiguity, and start a new `videos.insert` only with terminal non-creation evidence. Require Channel Binding Profile matches, private/`notifySubscribers=false`, and durable response video ID. Run API verification plus `roy-studio-verify` under its isolated read-only enforcement profile and least proven viewer role. Deliver PASS and playable claim-only WARN links together; never browser-upload, dispute, modify, promote, or delete automatically. A directly following `ok` on the sole complete Delivery Review selects visible hashes/video IDs without changing visibility. Revision requests create a new idempotent Publish Intent/video ID; superseded uploads stay private until an exact separately approved Remote Cleanup Plan is executed by the Privileged YouTube Executor. Continue runnable tracks and keep retention/recovery gates independent.
